# TL;DR
- 論文の内容を理解し、設計→解析→生成の3段階で依存関係を考慮したリポジトリを自動生成するマルチエージェントLLM基盤「PaperCoder（Paper2Code）」を提案。
- 著者公開実装をゴールドとして、モデル評価＋著者による人手評価で高品質・忠実性を実証。新ベンチマーク PaperBench でも強ベースラインを大幅上回る。

# 背景/貢献
- 背景: ML論文と対応コードの非公開/不足が再現性とフォローアップを阻害。
- 機運: LLMが科学文書理解とコード生成で有望、体系化で論文→実装の摩擦を低減可能。
- 貢献:
  - マルチエージェント枠組み「PaperCoder」を定義（計画/解析/生成の3フェーズ）。
  - 設計図（アーキ図）、ファイル依存、設定ファイルの自動生成と、モジュール分割・依存順序を意識したコード生成。
  - モデルベース＋人手（著者）評価で忠実性/品質を検証。PaperBenchで強ベースラインを大幅に上回る。

# 方法/設定
- フェーズ分割:
  - 計画: 高レベルロードマップ、システム構成図、依存グラフ、設定ファイルを作成。
  - 解析: 実装に必要な詳細（ハイパラ、入出力、訓練/評価手順）を論文から抽出。
  - 生成: 依存順にモジュール化されたコードを生成（ファイル/パッケージ単位）。
- エージェント設計: 各フェーズに特化したエージェント群で協調（役割分担・検証ループを設計）。
- 評価: 著者公開リポジトリがある場合はゴールド、モデル指標＋著者評価で比較。PaperBenchで総合比較。

# 結果/制約
- 結果:
  - 著者公開実装を参照した比較で、高品質かつ忠実度の高い実装を生成。
  - PaperBenchで既存強ベースラインに対し一貫して大幅優位。
- 強み: 文献理解→設計→依存考慮生成の分業で破綻低減。モジュール化により保守/差分修正が容易。
- 制約/注意:
  - 論文の曖昧記述・省略に弱い（外部ソース/RAG/補助常識の必要性）。
  - 実行環境/依存ライブラリの差分で動作不一致が起こりうる（CI/テスト補強が鍵）。
  - 著者評価はバイアス可能性、PaperBenchのドメイン分布/採点基準の汎化に留意。

# 気づき/疑問
- 実務転用: テンプレ化＋テスト駆動（最小実行例/CI）と組み合わせると再現性向上に寄与。
- 依存グラフ駆動の生成順序は他のコードエージェント（SWE系）にも有効。逆依存の検出/分解はどう扱うか。
- 評価: 機能テスト/数値一致/統計再現の3段階指標を分けると妥当性が上がりそう。
- 拡張: 追加データ（補遺/著者スライド/Issue）や自動FAQ生成で曖昧箇所を埋めるループの統合。
- ライセンス/倫理: 論文コードの生成物配布と元論文/データセットのライセンス整合性をどう担保するか。
