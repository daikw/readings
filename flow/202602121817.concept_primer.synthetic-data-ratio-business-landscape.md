# コンセプト概要
- 合成データ比率は重要だが、単独KPIとして最適化すると失敗しやすい。
- 実務では比率そのものより、どの工程で使うかと実データのアンカー維持が成果を左右する。
- ロボット基盤モデルと世界モデルでは合成データ活用が拡大しているが、主流は実データ置換ではなく実データ増幅である。

# 中核アイデア
## 対象読者
- ML/ロボティクス事業の意思決定者、データ戦略担当、研究リード、投資/DD担当。

## 想定利用シーン
- 事業計画、データ戦略策定、合成データ導入判断、投資判断、パートナー選定。

## 判定基準（実務での評価軸）
- 有効カバレッジ: 実環境の失敗モード、長尾、禁則（安全上の禁止動作）まで含めた網羅性。
- 実環境評価: 実ユーザ/実機/実センサで固定した評価セットを持ち、合成の改善と分離して計測できる。
- 継続的な実データ注入: “新鮮な実データ”が継続投入できる（固定比率でも腐るため、更新頻度が重要）。

## 合成比率の扱い（実務ルール）
- 合成比率は用途別に分ける（学習用、評価用、デバッグ/再現用、共有/テスト用）。
- 本番評価は原則「実データのみ」で持つ。合成評価は別指標として併記する。
- 目安レンジは“仮説”であり、最適化対象は比率ではなく上の3評価軸に置く。

## 事業カテゴリ別分析
### カテゴリA: データ収集代行/アノテーション（Human-in-the-loop）
- 何を売るか: 現地/実環境でのデータ取得能力、運用設計（手順・安全・QA）、ラベル品質、納期、守秘。
- 合成の位置付け: 長尾補完、ラベル補助、検証用の再現環境づくり（主役は実データ）。
- 合成比率の目安（仮説）: 学習用は10-30%程度から試し、長尾補完で最大50%近くまで上げることはある（タスク依存）。評価用は0-5%以下（原則は実評価を別立て）。
- コスト削減効果（実務観点）: 公表値は案件依存なので、再作業率、QA不合格率、1ラベルあたり工数、立ち上げ期間で計測するのが現実的。
- 主要プレイヤー（例）: Scale AI、Appen、TELUS Digital、Sama、Labelbox（ツール寄り）。
- 典型リスク: クライアントが実データ取得の主導権を握るため、合成は補助以上になりにくい。

### カテゴリB: データ販売/ライセンス/データ市場（Data-as-a-Product）
- 何を売るか: データそのもの、更新保証、派生特徴量、利用権、契約条項、来歴保証。
- 合成の位置付け: PII/機微を避けた共有、社外提供、規制対応のために使われることが多い。
- 合成比率の目安（仮説）: 商品としての“合成データ単体”は成立するが、差別化は比率ではなく来歴・監査・更新に移る。
- 主要プレイヤー/場（例）: AWS Data Exchange、Snowflake Marketplace、Databricks Marketplace。
- 主要論点: 法務要件（GDPR/HIPAA/金融規制等）に対して、生成プロセス・監査ログ・品質保証レベルを説明できるか。
- 価格モデル（仮説）: 実データは更新/独自性プレミアムが乗りやすい。合成は用途別（テスト/共有/学習）で価格体系を分けることが多い。

### カテゴリC: 合成データSaaS/SDK（主に構造化データ、PII回避）
- 何を売るか: 構造化データの合成、匿名化/マスキング、漏洩リスク評価、オンプレ実行、監査ログ。
- 合成の位置付け: モデル学習より、開発/検証/共有（QA環境、社外共有、PoC）でROIが出やすい。
- 合成比率の目安（仮説）: テスト/共有用は合成比率を高くできるが、統計一貫性と再同定リスクの評価が必須。
- 主要プレイヤー（例）: Gretel、MOSTLY AI、Tonic、Synthesized。

### カテゴリD: モデルAPI/基盤モデル（Foundation Models）
- 何を売るか: API、オンプレ提供、ファインチューニング、評価、ガードレール、データ協業。
- 合成の位置付け: データ拡張、多様性確保、反事例生成、ツール使用/会話データ生成などで使われる。
- 合成比率の目安（仮説）: 事前学習は5-15%程度、instruction tuning/ポストトレは20-40%程度、評価は5%以下（実データ評価を別立て）。
- 効果が出やすい工程: 長尾補完、多様性確保、バイアス点検、回帰テスト。
- 効果が出にくい工程: 安全/アライメントを“合成だけ”で閉じる（実運用のフィードバックが不可欠）。
- 主要プレイヤー（例）: OpenAI、Anthropic、Google、Meta、Microsoft、Amazon、NVIDIA、Mistral、Cohere。
- 典型リスク: 再帰学習（model collapse）で自己参照化し、外部世界への接地が失われる。

### カテゴリE: 垂直SaaS型モデル事業（運用ループがある）
- 何を売るか: 業務フロー内の予測/生成、監査、権限、承認、運用ダッシュボード、継続学習。
- 合成の位置付け: 現場KPIで不足しているケースをピンポイント補完する“増幅装置”。
- 合成比率の目安（仮説）: まずは失敗モードに対する部分的合成から入り、実ログ注入率を落とさない設計を優先する。

### カテゴリF: 視覚/自動運転/衛星/医療向け 合成データ工場（3Dレンダリング/センサシム）
- 何を売るか: 3Dアセット、シナリオ生成、光学/センサ/物理シミュレーション、完全ラベル生成。
- 合成の位置付け: 危険/希少/高コスト条件の生成、ドメインランダマイズ、評価回帰テスト。
- 合成比率の目安（仮説）: 認識系では合成を高比率で使える場合があるが、実機アンカー（キャリブレーション）がないと頭打ち。
- 主要プレイヤー（例）: Rendered.ai、Applied Intuition、Parallel Domain、Synthesis AI、NVIDIA Isaac Sim/Replicator（基盤）。

### カテゴリG: ロボット基盤モデル（Embodied Foundation Models / Robotics FMs）
- 何を売るか: 事前学習済みポリシー/表現、データ生成/評価フレームワーク、実機適応のツール。
- 合成の位置付け: 実機データが高価なため、シミュレーション軌道の大量生成で“実データ増幅”するのが基本。
- 合成比率の目安（仮説）: 事前学習/プレトレは合成主体になり得るが、最後は実機データで接地する（特に接触・摩擦・破れ等）。
- 主要プレイヤー/基盤（例）: NVIDIA Isaac Sim/Replicator/Isaac Lab、Open X-Embodiment/RT-X、RLBench、MuJoCo。

### カテゴリH: 世界モデル/シミュレータ基盤（World Models / Digital Twin）
- 何を売るか: 生成可能な環境モデル、計画/制御の内側で使う推論基盤、シミュレーション運用基盤。
- 合成の位置付け: 生成量より制御可能性（条件付け/編集/再現）と長期整合性、sim2real耐性が勝負。
- 主要プレイヤー（例）: NVIDIA Cosmos、DeepMind Genie系、学術（World4RL/Ctrl-World）。

## 研究の流れ（方向性）
- 方向性1: 大量生成中心から、有益サンプル選別（Deliberate Practice的）とカリキュラム設計へ。
- 方向性2: 長期整合性と行動条件付き生成を強化し、方策学習の内側でworld modelを使う方向へ。
- 方向性3: 単一エンボディメント最適化から、クロスエンボディメント転移を意識したデータ設計へ。

# 典型的な落とし穴
- 比率目標だけを追い、実環境の失敗モードを再現できないデータ配合になる。
- 合成データ再帰学習により分布が自己参照化し、汎化性能が劣化する。
- 学習時指標は改善しても、現場KPIや安全KPIで逆行する。
- 法規制や監査要件に対して、データ来歴と生成プロセスの説明可能性が不足する。
- 合成データの低コスト性を過信し、実データ収集パイプラインへの投資を止めてしまう。

## 落とし穴と頻発カテゴリ（対応表）
| 落とし穴 | 頻発カテゴリ | 具体例 |
| --- | --- | --- |
| 法規制・監査対応不足 | B, C, F | 医療/金融で来歴説明ができず監査不合格、共有停止 |
| 実データ投資停止 | A, F, G | 合成で代替できると判断し収集停止、汎化/安全が劣化 |
| 実環境の失敗モード不足 | A, E, G | 現場特有の例外処理が学習されず、OODAが回らない |
| 再帰学習による自己参照化 | D, H | 合成を継ぎ足して分布が崩れ、外部接地が弱まる |
| 指標の取り違え | 全般 | 学習指標は改善したが、現場KPI（安全/品質/歩留まり）が悪化 |

# 代表例
## GR00T N1
- 位置付け: 少量の人デモから大規模合成軌道を生成し、ロボット学習を加速する商用基盤の提示。
- 合成活用: シミュレーション/データ生成を前提に、実データ単独より改善を報告。
- 差別化: シミュレーション基盤と学習基盤の統合、デモからの増幅設計。

## Cosmos
- 位置付け: 物理AI向けworld foundation modelとして、生成データとシミュレーション運用の商用基盤化を示す。
- 合成活用: world model的な生成を“開発基盤”として提供する方向性。
- 差別化: 条件付き生成、編集性、ワークフロー統合。

## Open X-Embodiment / RT-X
- 位置付け: 大規模な実機軌道を土台に汎化を狙うデータ主導路線の代表。
- 合成活用: 公開一次情報では、基盤は実データである点が強いシグナル。
- 差別化: 実機データの規模、エンボディメント横断の一般化。

## Gemini Robotics
- 位置付け: internet-scale知識と実世界行動データの接地を重視する統合路線。
- 合成活用: 詳細は報告依存だが、評価・失敗ケース生成などで合成が入り得る。
- 差別化: マルチモーダル基盤モデルとの統合、実機接地の設計。

## Model Collapse系研究
- 位置付け: 合成データ再帰学習での性能劣化リスクを定式化し、混合運用の重要性を示す。
- 実務的示唆: カテゴリD/Hでは最低限の実データ注入（freshness）を止めないことが本体。

## 次のアクション（判断フローの最小形）
1. 対象タスクの「失敗モード」を列挙し、実データで観測できるかを確認する。
2. 合成で補完可能な失敗モードだけを対象に、生成条件を仕様化する。
3. 実評価セットを固定し、合成の投入で改善する指標と分離して計測する。
4. 実データ注入率（更新頻度）を下げずに、合成比率を段階的に上げる。
